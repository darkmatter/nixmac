## `.ok()` discarding errors (all intentional)

All 108 `.ok()` usages are intentional:

- **env var reads** (`std::env::var(...).ok()`, `store::get_*(...).ok().flatten()`): env vars and store values are optional fallbacks with `or_else` chains.
- **readdir `.filter_map(|e| e.ok())`**: skip unreadable entries; standard idiom for directory listing.
- **`serde_json::to_string(&x).ok()`** in watcher change detection: if serialization fails comparison, we emit the event conservatively (not a bug).
- **`summarize::find_existing::...ok()`** in `evolution.rs`/`watcher.rs`: change map is optional UI decoration; failure means "show empty map".
- **`db::get_db_path(app).ok()`**: DB is optional in some code paths (before initialization).
- **`evolve_state::get(app).ok()`**: state may not exist yet on first run.
- **`build_state::get(app).ok()`**: same.
- **`git::status(...).ok()`** in non-critical paths: git status is best-effort in background ops.
- **`parsed.ok().context(...)`** in edit_nix_file: converts `Parse<T>` (not a `Result`) to `Option` then to `Result` — this is the correct idiom for the `rnix` API.
- **`CL100K_BPE: Lazy<Option<CoreBPE>>`**: tokenizer init is fallible; `None` triggers the char-count fallback.
